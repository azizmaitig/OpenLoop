## T1: Spec-Driven Development (SDD) — Score: **1/3**

### ✅ What IS implemented

- **ADRs as architectural specs**: 11 ADR files under `docs/adr/` document every significant decision (ADR-0001 through ADR-0011). Each ADR follows the full format: Status, Context, Decision, Consequences, Alternatives.  
  - `docs/adr/0002-plan-driven-execution.md` — spec for the plan-driven execution mode
  - `docs/adr/0011-heal-wiring.md` — precise surgical spec mapping heal wire-up (types.ts → plan-executor.ts → execute-phases.ts)
  - `docs/adr/0010-stage-pipeline.md` — speculative spec for self-advancing pipeline (Path B)

- **Plan YAMLs as task specs**: 32 `.plan.yaml` files define tasks with scoped concerns. Example: `plans/notes-app-build.yaml` defines a 10-stage build with distinct `id` per task, timeout, and heals.

- **Plan-Writing-Guide**: `PLAN-WRITING-GUIDE.md` (284 lines) is a detailed pre-authoring guide that enforces schema, anti-patterns, and a pre-flight checklist before any plan is written. Referenced from `AGENTS.md` line 27-28.

- **Pre-implementation guardrails in plan YAMLs**: The `build-app-pipeline.yaml` defines safety rails R1-R5 as explicit comments directly in the plan template (lines 18-28).

### ❌ What is NOT implemented

- **No pre-implementation spec artifact as a FIRST-CLASS ENTITY**: The spec phase in `build-app-pipeline.yaml` (task `planning`, line 37-48) writes a spec to `.build/spec.md`, but there is no formal schema validation, no spec review gate before code generation, and no versioned spec lifecycle. The spec is a free-form markdown file generated by an LLM, not a structured, machine-parseable artifact.

- **No spec-review-before-code gate**: The `design-critique` phase (line 75-89) reviews the design against the spec, but this is an LLM advisory judgment, not a hard gate. The `design-critique` LLM prompt says `passed: false` does NOT fail a phase (CONTEXT.md line 321: "LLM verdicts are advisory only").

- **No spec diffing or change tracking**: Specs are written once and overwritten. No mechanism to compare spec revisions across iterations.

- **No GitHub Spec Kit or equivalent**: No `.github/ISSUE_TEMPLATE/` with spec templates, no OpenSpec integration.

### 🎯 Recommendations

1. **HIGH**: Make the spec artifact machine-parseable. Define a `SpecDef` in `src/types.ts` with structured fields: `goal`, `scope`, `nonGoals`, `acceptanceCriteria`, `techStack`. Parse LLM-generated specs into this shape. This enables programmatic spec-review gates.  
2. **MEDIUM**: Add a `spec-review` phase BEFORE `design-critique` that is a HARD gate (exit-code based). The phase runs a script that validates the spec against a schema and exits non-zero on missing fields.  
3. **LOW**: Integrate with OpenSpec (`.opencode/openspec/`) for spec lifecycle tracking across sessions.

## T2: Multi-Phase Pipelines — Score: **2/3**

### ✅ What IS implemented

- **10-stage pipeline decomposition**: `.omo/plans/build-app-pipeline.yaml` (169 lines) defines a complete pipeline:  
  `read-state → planning → research → design → design-critique → code → test → review → evaluate → verify`  
  Each stage has exactly one concern.

- **Narrow phase scope**: Each phase has a clearly bounded responsibility. The `code` phase prompt explicitly lists NON-GOALS (line 99-101): "do not add features absent from spec.md, do not add auth/payments/secrets handling..."

- **Phase-level guardrails**: Every phase has `timeoutMs` (ranging from 5s to 300s), and the `code` phase has `healCommand` + `maxRetries: 3` (lines 105-109). CONTEXT.md documents the per-phase guardrail model (section "Configuration", lines 215-228).

- **Sequential execution model**: Explicitly documented and reaffirmed — CONTEXT.md line 17: "All phases remain sequential (reaffirms ADR-0002)." State machine in CONTEXT.md (lines 163-174) shows `init → run → verify → done` with LOOP back to init.

- **Plan-driven mode**: `--plan <path>` CLI flag (loop-runner.ts:169-179) loads YAML tasks as phases. The `beforeLoop` hook (plan-executor.ts:21-51) maps tasks to PhaseDef[].

### ❌ What is NOT implemented

- **No self-advancing pipeline**: ADR-0010 (Path B) proposes a `StageManager` meta-plan with `onPass`/`onFail` transitions, per-stage retry, and `jump:<stage>` short-circuits. This is explicitly labelled as "Proposed (design only). Not yet implemented" (ADR-0010 line 4).

- **No cross-stage product state**: The only coordination between phases is `.build/*.md` files. There is no typed, versioned cross-stage state object. ADR-0010 calls this out: "no cross-stage product state beyond the `.build/*.md` files" (line 17-18).

- **No parallel execution**: Explicitly ruled out by ADR-0002. Real plan YAMLs are sequential chains. However, true multi-phase pipelines benefit from parallel branches where phases are independent (e.g., `research` + `design` could overlap if they don't depend on each other).

- **No dependency graph**: Phase dependencies are hardcoded in array order. No DAG or topological sort.

### 🎯 Recommendations

1. **MEDIUM**: Implement the StageManager from ADR-0010. Start with the minimal viable version: a `pipeline.yaml` that defines stages with `onPass`/`onFail` transitions, without the speculative `plan-generator`. This unlocks per-stage retry without full rebuilds.  
2. **LOW**: After StageManager, add parallel phase execution for independent branches. Begin by adding an `after` field to stage definitions (similar to `needs` in GitHub Actions) and a topological sort in `executePhases()`.

## T3: Context Engineering — Score: **2/3**

### ✅ What IS implemented

- **AGENTS.md** (32 lines): Loaded automatically. Defines loop mode (L1/L2), safety rules, verification requirements, plan authoring rules. References `PLAN-WRITING-GUIDE.md` and `STATE.md` explicitly.

- **CONTEXT.md** (321 lines): Comprehensive architecture document covering: design decisions, module-by-module breakdown, data flow (14-step sequence), state machine, configuration reference, v7/v8 improvements, plugin system, test strategy, quick start, and glossary with 7 defined terms.

- *(Consolidated into `AGENTS.md` — see Safety/guardrails section for constraints, Budget section for limits, Active loop configuration for loops.)*

- **PLAN-WRITING-GUIDE.md** (284 lines): Detailed plan authoring reference with mental model, schema, field reference, LLM contract, guardrails, two plan archetypes (6A/6B), anti-pattern table (13 items), and pre-flight checklist (11 items).

- **STATE.md** (10 lines): YAML frontmatter with `last_run`, `current_state`, `iteration`, health metrics.

- **11 ADRs**: All follow the standard format — Context, Decision, Consequences, Alternatives.

- **5 SKILL.md files** with YAML frontmatter: `minimal-fix`, `loop-verifier`, `loop-triage`, `loop-constraints`, `loop-budget`.

### ❌ What is NOT implemented

- **No .claude/ or .agents/ directories**: These are standard locations for agent-readable context across different agent platforms (Claude Code, OpenCode, etc.). Their absence means the project can't leverage cross-platform context injection.

- **No slash commands**: No `/pr`, `/review`, `/migrate`, `/spec`, `/plan` commands defined in the opencode.json config. The `agent` definitions in `opencode.json` (loop-triage, implementer, verifier) are agents, not slash commands.

- **No clean layered context model**: While docs exist, they are flat files. There is no structured layering of system → project → task → code → tools → memory as recommended by modern context engineering.

- **No skill auto-loading by topic matching**: Skills are loaded via the `agent` definitions in `opencode.json`, not automatically when the conversation topic matches the skill description.

- **CONTEXT.md has stale content**: Line 321 says "healAndRetry is defined as a recovery variant but currently has no live caller" — this is FALSE in the current codebase (execute-phases.ts line 108 calls healAndRetry). The glossary entry contradicts the implementation.

### 🎯 Recommendations

1. **HIGH**: Create `.claude/` directory with a `CLAUDE.md` summarizing the project. This is the standard entry point for Claude Code and other agents. Cross-reference `AGENTS.md` and `CONTEXT.md` from it.  
2. **MEDIUM**: Fix stale documentation. Audit `CONTEXT.md` for claims that don't match the source (especially the heal wiring claim at line 321 and "plan mapping never piped healCommand/maxRetries through" at line 320).  
3. **MEDIUM**: Add slash commands to `opencode.json`: `/spec`, `/plan`, `/review`, `/triage` as quick-invoke patterns that load specific skill files.  
4. **LOW**: Implement a structured context layering approach where each layer (system → project → task → code → tools → memory) is a separate file loaded incrementally as the session progresses.

## T4: Adversarial Review — Score: **1/3**

### ✅ What IS implemented

- **Maker-checker plugin**: `src/maker-checker-plugin.ts` (111 lines) implements a dual-phase maker/checker pattern. The `onPhaseEnd` hook intercepts maker pass → injects a checker phase → evaluates with LLM → retries on fail up to `maxCheckerRetries` (default 2).  
  - File: `src/maker-checker-plugin.ts:58-94`

- **loop-verifier SKILL.md**: Defines an independent verifier persona whose job is to "reject unless evidence is strong" (line 12). Has 5-point checklist: scope, intent, tests, no cheating, risk assessment. Output format requires APPROVE/REJECT/ESCALATE_HUMAN verdict with evidence.  
  - File: `skills/loop-verifier/SKILL.md:23-47`

- **AGENTS.md mandates verifier for L2+**: Line 21: "For L2+ changes, dispatch a verifier sub-agent after implementation."

- **opencode.json defines verifier agent**: Line 28-37 — a subagent with `deny` edit permission, configured to review diffs and test evidence.

- **LLM evaluation as advisory**: CONTEXT.md explicitly documents that LLM judgments ARE NOT authoritative (R4 Evidence-not-claim, line 25-27). `loop-runner.ts:120-125` — `resolveHardcoded` only uses exit code, not LLM passed/false.

### ❌ What is NOT implemented

- **Maker-checker is OPT-IN**: The plugin requires explicit `--plugins` flag or config entry. It is NOT loaded by default and NOT documented in `opencode.json`. There is no "always on" adversarial review for every phase.

- **No multi-consensus review**: No mechanism for 2+ independent reviewers (e.g., 3 Oracle agents voting). The maker-checker is a single review step, not an ensemble.

- **No non-LLM verification alongside LLM review**: The review phases (design-critique, review, evaluate) are all LLM-based. No static analysis, no linter, no type-checker runs alongside the LLM to catch things the LLM misses.

- **No evidence-based review requirement**: The verifier skill says "run tests" but there is no enforcement that the implementer produced evidence (screenshots, test logs, type-check output) that the verifier must cite.

- **No reviewer blind-spot acknowledgment**: No documentation of what the LLM reviewer is likely to miss (e.g., race conditions, security vulnerabilities, performance regressions in edge cases).

### 🎯 Recommendations

1. **HIGH**: Make maker-checker ON by default for all L2+ plan-driven runs. Remove the `enabled` flag or default it to `true`. The dual-phase pattern should be the standard, not an option.  
2. **MEDIUM**: Add a second independent reviewer phase. Even a simple "run the same evaluation twice with different LLM configs" catches the blind-spot problem. Store both judgments in state and require consensus for pass.  
3. **MEDIUM**: Add a non-LLM review phase: run a linter (`bun run lint`), a type-checker (`bun run tsc --noEmit`), and a security scanner alongside the LLM review. Fail the phase if any non-LLM check fails.  
4. **LOW**: Add evidence-citation requirement to the verifier: the implementer must produce artifact paths (e.g., `_agent-loop-output/phase-test.log`), and the verifier must reference them by path in the verdict.

## T6: MCP Servers — Score: **1/3**

### ✅ What IS implemented

- **MCP execution module**: `src/mcp.ts` (122 lines) implements full JSON-RPC 2.0 dispatch over stdin/stdout.  
  - Line 59-64: Builds valid JSON-RPC request with `jsonrpc: '2.0'`, `method: 'tools/call'`, `params: { name, arguments }`.  
  - Lines 52-57: Handles prompt as JSON (valid JSON → use as args) or plain string (wrap in `{ prompt }`).  
  - Lines 67-96: `executeMcpPhase()` spawns MCP subprocess, sends request, parses response.

- **Timeout handling**: `executeWithTimeout` from `safety.ts` (AbortController) is used at line 67 and 96. `PhaseTimeoutError` at line 101-109.

- **Graceful degradation**: Every error path returns a structured `PhaseResult` with `status: 'error'` — never throws. Lines 24-44, 97-121.

- **MCP used for LLM evaluation path**: `evaluate.ts` (lines 55-79) has an MCP path that constructs an eval phase and calls `executeMcpPhase`.

- **JSON-RPC transport module**: `src/json-rpc.ts` handles the low-level send/receive (referenced in mcp.ts line 12).

### ❌ What is NOT implemented

- **No MCP server definitions in the project**: The project is purely an MCP consumer. No MCP servers are defined. Every MCP server it uses (agentmemory, GitHub MCP, etc.) is external.

- **No tool discovery mechanism**: The project hard-codes tool names. No `tools/list` call is made to discover available tools. If the MCP server changes its tool set, the project breaks.

- **No reconnect logic**: If the MCP subprocess crashes or disconnects, there is no retry or reconnect. The error surfaces immediately as a phase failure.

- **No MCP server health checking/ping**: No periodic `ping` or health check to verify the MCP server is still alive before sending a request.

- **No streaming support**: All MCP calls are request-response. No support for MCP's streaming notifications (e.g., progress updates).

### 🎯 Recommendations

1. **HIGH**: Add tool discovery via `tools/list` before first use. Cache the result alongside the phase config. If a tool is not in the advertised list, fail early with a clear message instead of sending an invalid request.  
2. **MEDIUM**: Implement automatic reconnect for MCP subprocess crashes. Wrap `executeMcpPhase` in a retry loop that re-spawns the server on connection failure, up to 2 retries with exponential backoff.  
3. **MEDIUM**: Add a `ping` call before every `tools/call` for long-running phases (>30s timeout). This prevents failing after waiting minutes for a dead server to time out.  
4. **LOW**: Add optional streaming support for progress updates during long MCP tool calls (code generation, research phases).

## T7: Verification Gates + Evidence — Score: **2/3**

### ✅ What IS implemented

- **Exit-code hard gate**: `execute-phases.ts` line 68 runs `executeShellCommand` and checks `exitCode !== 0` at lines 94-103. This is the authoritative pass/fail mechanism. CONTEXT.md documents this as "R4 Evidence-not-claim" (line 25-27 of build-app-pipeline.yaml).

- **LLM judgment as advisory**: CONTEXT.md line 321: "LLM verdicts are advisory only." `loop-runner.ts:120-125` (`resolveHardcoded`) only uses exit code for pass/fail. LLM `passed: false` does not fail a phase.

- **Checkpoint after every phase**: `execute-phases.ts` lines 130-161 save a `.checkpoint.json` after EVERY completed phase. This is the authoritative crash recovery source.

- **Phase-level evidence logging**: Each phase result carries `stdout`, `stderr`, `exitCode`, and `durationMs`. `execute-phases.ts` lines 165-181 write structured `RunLogEntry` to `loop-run-log.md` with `run_id`, `pattern`, `outcome`, `duration_ms`.

- **Memory hooks**: `memory-hooks.ts` logs health scores (line 183-186), archives sessions (line 193), and saves failure lessons (line 150). All fire-and-forget.

### ❌ What is NOT implemented

- **`evidencePath` field is ALWAYS empty**: `PhaseResult.evidencePath` (types.ts:53) is defined but NEVER populated. In `execute-phases.ts` line 206, it's hardcoded to `''`. In `mcp.ts` line 80, it's `''`. This field is dead code.

- **No screenshots, curl traces, or type-check output**: No non-text evidence is ever captured. For UI projects, this means visual regressions can't be proven or disproven.

- **No "evidence alongside code"**: Evidence (test logs, build output) is ephemeral — it lives in `loop-run-log.md` as JSON entries, not alongside the changed files. There's no `.build/evidence/` directory where each phase deposits its evidence artifacts.

- **Verify phase delegates to opencode run**: The `verify` phase in `build-app-pipeline.yaml` (lines 151-158) runs `opencode run "..."` to verify — meaning verification itself goes through an LLM sub-agent. The "non-LLM gate" is enforced only by the exit code of the opencode subprocess.

### 🎯 Recommendations

1. **HIGH**: Wire `evidencePath` — have `executeShellCommand` and `executeMcpPhase` save stdout/stderr to a unique file under `_agent-loop-output/evidence/<phase-name>-<timestamp>.log` and populate `evidencePath`. This makes evidence persistent and referencable.  
2. **MEDIUM**: Add a dedicated `verify` command that runs WITHOUT an LLM sub-agent. For a Bun/TS project, this means `bun run build && bun test` directly via `executeShellCommand`, not via `opencode run`. The LLM should never be in the verify path for non-LLM gates.  
3. **LOW**: Add support for binary evidence capture: if a phase produces a screenshot, screenshot path, or other binary output, store the path in `evidencePath` alongside text evidence.

## T8: Child Session Isolation — Score: **2/3**

### ✅ What IS implemented

- **Git worktree isolation**: `src/worktree.ts` (128 lines) implements full worktree lifecycle:
  - `createWorktree()` (line 23-35): Creates a git worktree with branch name sanitization
  - `runInWorktree()` (line 40-44): Runs commands in the worktree directory
  - `verifyInWorktree()` (line 84-90): Runs tests and returns pass/fail
  - `discardWorktree()` (line 53-79): Removes with graceful handling of dirty/not-found states
  - `pruneStaleWorktrees()` (line 101-127): Cleans old worktrees exceeding limit

- **AGENTS.md mandates worktree for code changes**: Line 16: "Use a git worktree for every code-changing attempt."

- **File-based context between phases**: `build-app-pipeline.yaml` defines the pattern: each phase writes its output to a `.build/*.md` file, and the next phase reads that file (R2 Context-rot guard). This prevents cross-phase context bleed.

- **Fresh subprocess per phase**: Each phase in plan YAMLs runs as a separate `opencode run` subprocess, not as a continuation of the previous LLM session.

- **Worktree history**: `.omo/worktrees/dashboard-loop-state/` shows an existing worktree with its own `STATE.md`, AGENTS.md (containing constraints and budget), ADRs, and plans — indicating worktrees carry full project context.

### ❌ What is NOT implemented

- **No container-level isolation**: Worktrees isolate git state but not OS-level resources (environment variables, filesystem, network). No Docker, no sandbox, no chroot.

- **No explicit environment isolation per phase**: The `RunOptions` in `shell.ts` (line 27-32) has `cwd` and `timeoutMs` but no `env` field for per-phase environment variables.

- **No process-level isolation**: All phases share the same process tree. A runaway subprocess in one phase can affect others (e.g., memory exhaustion in the `code` phase crashes the `verify` phase).

- **No clean environment for daemon triggers**: When the orchestrator enqueues a task via trigger (orchestrator.ts:150), there's no environment cleanup between trigger firings.

### 🎯 Recommendations

1. **MEDIUM**: Add an `env` field to `RunOptions` in `shell.ts` so each phase can set/override environment variables independently. Update `PhaseDef` to optionally carry `env` overrides.  
2. **MEDIUM**: For daemon mode, add process group isolation. Use `Bun.spawn` with process group IDs so killing one phase's subprocesses doesn't affect others.  
3. **LOW**: Add an optional `sandbox` mode using Docker containers for security-critical phases (e.g., running untrusted code). This is a stretch for most use cases but worth documenting as a future option.

## T9: Heal Commands + Iteration Loops — Score: **2/3**

### ✅ What IS implemented

- **healCommand + maxRetries on plan YAML tasks**: `types.ts` lines 13-15 define `healCommand?: string` and `maxRetries?: number` on `PhaseDef`. `plan-executor.ts` lines 30-31 map these from YAML tasks to phases.

- **HealAndRetry implementation**: `src/recovery.ts` lines 108-130 implement `RecoveryStrategy.healAndRetry`:
  - Runs `healCommand` up to `maxRetries` times (line 115: `for (let attempt = 1; attempt <= heal.maxRetries; attempt++)`)
  - On heal success (exit 0), re-runs the original `phase.command` (line 118)
  - If the retry exits 0, mutates the result to "pass" in place (lines 119-126)
  - Returns `{ healed: true/false }`

- **Heal is wired in execute-phases.ts**: Lines 108-127 — when a phase fails AND has a healCommand, the code calls `RecoveryStrategy.healAndRetry` with the phase and result. This code is LIVE (NOT dead code despite ADR-0011 being labeled as "spec").

- **maxIterations hard cap**: `config.ts` line 20 enforces Math.min with cap 20. `safety.ts` lines 73-78 implement `checkMaxIterations`.

- **Example in real plan**: `plans/notes-app-build.yaml` lines 71-76: `code` task has `healCommand` and `maxRetries: 3`. The heal re-invokes opencode with the failing build output and code.diff.

- **Checkpoint resume**: `loop-runner.ts` lines 139-165 implement checkpoint detection with user prompt (Y/n). `checkpoint.ts` (52 lines) handles save/load/clear/filter.

### ❌ What is NOT implemented

- **No error categorization (transient vs terminal)**: Every failure gets the same heal treatment. There's no mechanism to detect that a failure is permanent (e.g., "module not found" → pointless to retry) vs transient (e.g., "network timeout" → worth retrying).

- **No backoff between retries**: Retries fire immediately with no delay. If the failure is race-condition-induced, immediate retries will hit the same condition. Exponential backoff or jitter would improve success rates.

- **No retry budget tracking**: There's no cross-phase retry budget. If every phase retries 3 times, a 10-phase plan could run 30 retries. The maxIterations cap doesn't limit internal retries within a single iteration.

- **No heal logging to evidence**: Heal attempts are not logged to `evidencePath`. If a heal succeeds, the evidence only shows the final pass, not the failure + heal path. This makes debugging heal patterns difficult.

- **CONTEXT.md is stale about heal status**: Lines 320-321 claim heal has no live caller, which contradicts the code at execute-phases.ts:108.

### 🎯 Recommendations

1. **MEDIUM**: Add error categorization in `recovery.ts`. Define a simple heuristic: errors matching common transient patterns (ECONNREFUSED, ETIMEDOUT, ENOSPC, "process exited with signal") get retried; everything else is terminal.  
2. **MEDIUM**: Add exponential backoff between heal retries. Start at 1s, double each attempt, cap at 30s. This is a 5-line addition to the `healAndRetry` for-loop in `recovery.ts:115`.  
3. **MEDIUM**: Fix CONTEXT.md to accurately reflect the current heal wiring. The document's claims about "no live caller" are false and will mislead future developers.

## T10: Boring Stack Rule — Score: **2/3**

### ✅ What IS implemented

- **Mainstream runtime**: TypeScript on Bun is a widely-used, well-documented combination with excellent AI training data coverage. Bun's API is compatible with Node.js, so standard Node.js patterns work.

- **Minimal production dependencies**: `package.json` shows only ONE runtime dependency: `js-yaml ^5.2.1`. Everything else is dev dependencies for type definitions and TypeScript itself.

- **Standard test framework**: `bun test` (built-in, Jest-compatible API). 79 test files with 984 expect() calls (per CONTEXT.md line 251).

- **Standard project structure**: Conventional `src/` structure with barrel export (`index.ts`), separate `__tests__/` directory.

- **No exotic patterns**: The state machine is a flat lookup table (state-machine.ts:4-9), not a library. Plugins are simple interface implementations (plugins.ts:5-12). No dependency injection framework, no inversion of control container.

### ❌ What is NOT implemented

- **No TypeScript strict mode in tsconfig**: Not verified during this audit (tsconfig.json not read), but CONTEXT.md doesn't mention strict mode, which is standard for production TypeScript projects.

- **No linting setup**: No ESLint, Biome, or other linter configured. `package.json` has no lint script. The only verification is `bun test`.

- **No Postgres/Tailwind/shadcn**: While these are not needed for a loop orchestrator (it doesn't have a UI or persistent storage), the "boring stack" recommendation typically includes these for full-stack projects. The project doesn't NEED them, but the absence means no established patterns for when they ARE needed.

- **Bun-specific APIs**: The project uses `Bun.spawn`, `Bun.file`, `Bun.write`, `Bun.readableStreamToText` — these are Bun-specific and not available in standard Node.js. This is a mild lock-in.

### 🎯 Recommendations

1. **LOW**: Add a linting setup. Biome would be the natural choice for a Bun/TypeScript project — it's fast, has a single binary, and covers both formatting and linting. Add `"lint": "biome check src/"` to `package.json`.  
2. **LOW**: Ensure `strict: true` in `tsconfig.json` (or enable it if not set). The project already has good type discipline based on the source code reviewed.  
3. **INFO**: The Bun-specific APIs are a reasonable trade-off for a project that's already Bun-native. Document the Bun dependency explicitly in README/AGENTS.md so contributors know.
